### Rules ###

#### General Output Rules ####
- Output Format: Return ONLY a valid JSON object. No explanations, no markdown code blocks, no extra text before or after the JSON.
- Schema: Do not add or omit any keys from the schema.
- Default to Null: If a value for a key is not present in the text, its value in the JSON must be `null`.
- Mandatory Empties: The keys `URL` and `Aggregated` must be set to an empty string `""`, regardless of the input text.
- Jurisdiction County: Always set to "Alexandria". Do not extract from the document.
- Instrument Number = Case Number: Extract the case number from the document and put it ONLY in "Instrument Number". Do NOT put the case number in any other field (Document ID, Instrument Duplicate Prompt, or elsewhere).

#### Address/Location Rules ####
- Standard Addresses (Subscriber, POW, Administrator, Executor, Applicants, Personal Representative, Heirs, Attorney, Decedent Address in party information section): Split into exactly 4 separate fields:
  - "Address": street only ("5008 Just Street, NE")
  - "City": city name ("Washington")
  - "State": state abbreviation
  - "Zip": zip code ("20019")
  Do NOT include city, state, or zip in the Address field.

#### Name Processing Rules ####
- Names (Decedent, Subscriber, POW, Administrator, Executor, Applicants, Personal Representative, Attorney, Heirs):
  1. Capitalization: First letter uppercase, rest lowercase ("ANDERSON" → "Anderson", "o'brien" → "O'Brien")
  2. Remove ALL commas and special characters, keep only spaces between words
  3. Company/organization override: If name contains any of these as STANDALONE WORDS (not part of another word), treat as company:
     - LLC, Inc, Corp, Ltd, LLP, LP, PLC, PLLC, PA, PC (with or without periods)
     - "Co.", "Company", "Corporation", "Incorporated"
     - "Department", "Agency", "Bank", "Trust"
     - Starts with "Estate of"
     Put FULL company name in "First Name" and set "Last Name" to null.
     Example: "NewRez LLC d/b/a Shellpoint" → First Name: "Newrez Llc D B A Shellpoint", Last Name: null
     Note: "O'Connor" or "McCoy" are NOT companies (Co is part of the word, not standalone)
  4. Person names: Remove ALL middle names/initials and suffixes (Jr, Sr, II, III, IV, V, Esq, MD, PhD, etc.). Output ONLY one first name and one last name.
  5. Examples:
     - "Anderson, Jack Michael Jr" → First: "Jack", Last: "Anderson"
     - "Van Der Berg, John David Sr" → First: "John", Last: "Van Der Berg"


#### Document Type Rule ####
- Document Type: Extract from the beginning of the document. Look for the document title or type near the top (e.g., "List of Heirs", "Order", "Petition", "Affidavit"). Use the title Case form. If not found, set to null.

#### Data Format Rules ####
- Date Format: yyyy/mm/dd (e.g., "2024/03/15")

- Party Arrays (Subscriber, POW, Administrator, Executor, Applicants, Personal Representative): Each key holds a list of person objects. Extract all matching parties from the document. Structure per person: {"First Name": null, "Last Name": null, "Address": null, "City": null, "State": null, "Zip": null, "Phone Number": null}
  - Subscriber: All subscribers found in the document
  - POW: All proponent(s) of the will
  - Administrator: All administrators
  - Executor: All executors
  - Applicants: All individuals in the Party Information section but no attorneys (use exact applicant name, address, phone number as found)
  - Personal Representative: All personal representatives (PR) found in the document
  Each person becomes one dictionary in the list. If none found for a key, return empty array [].

#### Role Independence Rule ####
- If a person appears under both roles (e.g., Applicant and Heir), include them independently in BOTH arrays.
- Each role must be populated strictly based on how the document labels that person.
- Do NOT merge Applicants into Heirs.
- Do NOT exclude Applicants because they are also Heirs.
- Do NOT assume role equivalence—treat each role separately.

#### Cross-Reference Party and Heir Data Rule ####
- When the same individual appears in both a party array (Subscriber, POW, Administrator, Executor, Applicants, Personal Representative) and in Heirs, match them by First Name + Last Name (case-insensitive).
- For matched individuals: fill any missing data points in one record using the other. For example, if an Heir has Address/City/State/Zip missing but the same person in Applicants has that data, copy it into the Heir record. Conversely, if a party has missing address but the Heir has it, copy into the party record.
- Apply this for: Address, City, State, Zip, Phone Number (where applicable). Only fill null/empty values; do not overwrite existing data.
- Keep both entries in their respective arrays—only enrich missing fields by cross-referencing.

- Heirs Array: Structure per heir: {"First Name": null, "Last Name": null, "Relationship": null, "Age": null, "Address": null, "City": null, "State": null, "Zip": null}
  If no heirs found, return empty array []

### Task ###
Parse the provided "TEXT DATA" and generate a single JSON object as the output.
